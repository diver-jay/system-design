# TIL: Database Indexing Deep Dive (2026-04-28)

## 1. B-Tree: "백과사전식 저장"
- **철학**: 제자리 쓰기 (In-place Update). 데이터의 정해진 위치(Page)를 찾아가서 덮어씀.
- **I/O 방식**: **Random I/O**. 매 쓰기마다 디스크의 다른 위치로 점프해야 하므로 쓰기 성능이 제한됨.
- **장점**: 읽기 성능이 매우 빠르고 일정함. 데이터가 항상 단 한 곳에 존재하기 때문.
- **주 사용처**: 관계형 DB (MySQL, PostgreSQL), 강력한 일관성과 읽기 성능이 중요한 시스템.

## 2. LSM-Tree: "포스트잇식 저장"
- **철학**: 일단 덧붙이기 (Append-only / Shadowing). 기존 데이터를 수정하지 않고 최신 데이터를 계속 위에 쌓음.
- **I/O 방식**: **Sequential I/O**. 메모리에 모았다가 디스크 끝에 순차적으로 쏟아붓기 때문에 쓰기가 압도적으로 빠름.
- **핵심 컴포넌트**:
    - **WAL (Write Ahead Log)**: 메모리(휘발성) 데이터 유실을 막기 위해 디스크에 먼저 남기는 기록.
    - **MemTable (SkipList)**: 메모리 내 자료구조. 정렬된 상태를 유지하며 삽입/검색을 $O(\log N)$으로 처리.
    - **SSTable (Sorted String Table)**: 디스크에 저장된 정렬된 데이터 파일. LSM-Tree 읽기의 최소 단위.
    - **Bloom Filter**: 특정 키의 존재 여부를 확률적으로 확인하여 불필요한 디스크 I/O를 방지.
    - **Compaction**: 여러 SSTable을 병합하여 중복 데이터를 제거하고 읽기 성능을 유지하는 정리 작업.
- **장점**: 대규모 쓰기 처리량(Throughput) 확보.
- **단점**: 읽기 증폭(Read Amplification) 및 컴팩션 부하(Compaction Stall).

## 3. 면접 핵심 포인트 (Staff Engineer Level)
- "쓰기가 많다면 LSM-Tree 기반을, 읽기가 많고 일관성이 중요하다면 B-Tree 기반을 선택하겠습니다."
- "LSM-Tree는 쓰기 성능을 위해 디스크 I/O 비용을 나중(Compaction)으로 미루는 구조입니다."
- "B-Tree의 랜덤 I/O 문제를 해결하기 위해 LSM-Tree는 모든 쓰기를 순차 I/O로 변환합니다."
