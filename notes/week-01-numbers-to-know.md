## 치트시트

| Component | Key Metrics | 샤딩/스케일링 고려 시점 |
|-----------|-------------|----------------------|
| **Cache (Redis)** | ~1ms 레이턴시, 100k+ ops/sec, 최대 1TB 메모리 | Hit rate < 80%, 레이턴시 > 1ms, 메모리 > 80% |
| **Database (RDS/Aurora)** | 50k TPS 읽기, 10-20k TPS 쓰기, 5-30ms 레이턴시, 64TiB+ 스토리지 | Write > 10k TPS, 스토리지 50TiB+ 근접, 지역 분산 필요 |
| **App Server** | 100k+ 동시 접속, 8-64 코어, 64-512GB RAM | CPU > 70-80%, 레이턴시 SLA 초과, 메모리 > 80% |
| **Message Queue (Kafka)** | 1M msgs/sec/broker, sub-5ms 레이턴시, 50TB 스토리지 | 처리량 800k msgs/sec 근접, consumer lag 증가 |

## 면접에서 흔한 실수

### 1. 너무 빠른 샤딩
Yelp 설계 예시:
- 비즈니스 10M개 × 1KB = **10GB** → 샤딩 불필요
- 리뷰 포함 10배: **100GB** → 여전히 단일 DB로 충분

LeetCode 리더보드:
- 100k 대회 × 100k 유저 × 40B = **400GB** → 단일 캐시로 커버 가능

### 2. SSD 레이턴시 과대 추정
인덱스 있는 단순 row lookup은 서브 밀리초~수 밀리초. 이미 충분히 빠름.
캐시 레이어를 추가할 필요가 없는 경우가 많다. (복잡한 쿼리는 예외)

### 3. 쓰기 처리량 과소 평가
- 잘 튜닝된 Postgres: **단순 쓰기 20k+ TPS** 처리 가능
- 5k WPS 보고 메시지 큐 바로 붙이는 건 오버 엔지니어링

메시지 큐가 필요한 경우:
- 장애 시 보장된 전달 필요
- 이벤트 소싱 패턴
- 20k+ WPS 초과
- 프로듀서-컨슈머 디커플링 필요

## 핵심 인사이트

- 단일 DB: **수십 테라바이트** 처리 가능
- 단일 캐시: **전체 데이터셋**을 메모리에 올릴 수 있음
- 메시지 큐: 레이턴시 5ms 이하 → 동기 흐름에도 사용 가능
- App 서버: 로컬 메모리 최적화 적극 활용 가능

**설계 원칙: 먼저 단순하게. 수치로 정당화한 후에 복잡성 추가.**
