# TIL: Caching Strategies & Architecture (2026-04-30)

## 1. Caching Strategies (Write Patterns)
- **Cache-Aside (Lazy Loading)**: 앱이 직접 DB에 쓰고 캐시를 **삭제(Delete)**. 
    - *Insight*: 실무 90% 점유. 캐시 장애 시 DB로 버틸 수 있는 '안전성'이 강점.
- **Write-Through**: 캐시와 DB에 동시 저장. 
    - *Insight*: "쓰기"는 느려도 되지만, 이후 발생하는 수만 번의 "읽기"를 위해 캐시를 미리 채워두는 전략. 강한 정합성 필요 시 사용.
- **Write-Back (Write-Behind)**: 캐시에 먼저 쓰고 DB는 나중에. 
    - *Insight*: DB 쓰기 부하를 획기적으로 줄임(100번 수정 -> 1번 저장). 유실 위험은 Message Queue 등으로 보완.

## 2. Decision Factors for Strategy
1. **성능(Latency/Throughput)**: Write-Back 최강.
2. **정합성(Consistency)**: Write-Through 최강.
3. **안전성(Availability)**: Cache-Aside 최강 (SPOF 방지).
4. **운영 복잡도**: Cache-Aside는 앱 로직 복잡, 나머지는 인프라 설정 복잡.
5. **DB 부하 절감**: Write-Back만 가능 (Batch Write).

## 3. Advanced Insights & Hybrid Architecture
- **Single-Flight vs Soft TTL**: 지연(Latency)을 허용할 것인가, 낡은 데이터(Stale)를 허용할 것인가의 UX 선택.
- **Hybrid Approach**: 
    - 데이터 중요도에 따라 한 서비스 내에서 여러 전략 혼용 (결제는 DB, 좋아요는 Write-back).
    - **Reliable Write-back**: Cache + Message Queue + DB 조합으로 성능과 안전성 모두 확보.
    - **Multi-tier Caching**: L1(Local) + L2(Global/Redis)로 Redis 부하까지 분산.
