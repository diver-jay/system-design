# 시스템 디자인과 지연 시간(Latency) 핵심 정리

구글 SRE(Site Reliability Engineering)에서 강조하는 '엔지니어가 반드시 알아야 할 지연 시간' 수치와 이를 활용한 시스템 설계 계산법을 정리한 문서입니다.

---

## 1. 주요 지연 시간 수치 (Latency Numbers)

컴퓨터 시스템의 각 계층에서 발생하는 작업별 소요 시간입니다. [cite: 1, 2]

| :--- | :--- | :--- | :--- |
| **L1 캐시 참조** | 1 ns | - | |
| **분기 예측 오류** | 3 ns | - | |
| **L2 캐시 참조** | 4 ns | - | |
| **뮤텍스 잠금/해제** | 17 ns | - | |
| **메인 메모리 참조** | 100 ns | - | |
| **SSD 4KB 무작위 읽기** | 20,000 ns | 0.02 ms | |
| **데이터센터 내 왕복** | 500,000 ns | 0.5 ms | |
| **HDD 순차 읽기 (1MB)** | 5,000,000 ns | 5 ms | |
| **디스크 탐색 (Seek)** | 10,000,000 ns | 10 ms | 가장 큰 병목 |
| **대륙 간 패킷 왕복** | 150,000,000 ns | 150 ms | 물리적 한계 |

---

## 2. 직관적인 시간 비교 (1ns = 1초 가정 시)

지연 시간의 격차를 실생활의 시간으로 변환하면 그 심각성을 쉽게 이해할 수 있습니다.

* **L1 캐시 참조 (1ns)**: **1초** (눈 깜빡임)
* **메인 메모리 참조 (100ns)**: **1분 40초** (커피 한 잔 준비)
* **SSD 무작위 읽기 (20,000ns)**: **약 5.5시간** (서울-부산 편도)
* **디스크 탐색 (10,000,000ns)**: **약 115일** (계절이 바뀜)
* **대륙 간 패킷 왕복 (150ms)**: **약 4.7년** (대학 학위 취득)

---

## 3. 지연 시간과 처리량(QPS) 계산법

### 기본 공식
단일 실행 단위(스레드)가 쉬지 않고 작업할 때의 최대 성과는 지연 시간의 역수입니다.
$$Throughput (QPS) = \frac{1}{Latency}$$

### 전체 시스템 QPS
$$Total QPS = \frac{1}{Latency} \times 병렬\ 스레드\ 수$$

---

## 4. 실전 설계 적용: 1M QPS URL 단축 서비스

초당 100만 건의 요청을 처리하기 위해 필요한 자원 예측 예시입니다.

1. **메모리 조회 (100ns)**
   * 단일 스레드 처리량: 10,000,000 QPS
   * **결론**: 단일 서버의 메모리 캐시만으로도 1M QPS 대응 가능.

2. **네트워크 호출 (500μs)**
   * 단일 스레드 처리량: 2,000 QPS
   * **결론**: 1M QPS를 달성하려면 최소 500개의 병렬 연결(Concurrency) 필요.

3. **디스크 DB 조회 (10ms)**
   * 단일 스레드 처리량: 100 QPS
   * **결론**: 1M QPS를 위해 10,000개의 독립적인 디스크 I/O가 필요함. 단일 DB로는 불가능하며 샤딩이나 캐싱이 필수적임.

---

## 5. 요약 및 설계 원칙 [cite: 1]

* **메모리는 디스크보다 10만 배 빠르다**: 성능이 중요한 데이터는 메모리에 둔다.
* **네트워크 왕복은 비싼 비용이다**: 데이터센터 내 통신은 0.5ms지만, 전 세계 대상 서비스는 CDN 등을 통해 지연 시간을 최소화해야 한다.
* **순차 읽기가 무작위 읽기보다 빠르다**: 디스크나 네트워크 전송 시 데이터를 모아서 처리하는 것이 유리하다.
